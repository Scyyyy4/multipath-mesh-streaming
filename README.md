## System Overview

The project evolved through two platforms:

| Platform | Description |
|---|---|
| **Raspberry Pi Testbed** | 5 × Raspberry Pi 4B nodes, multi-hop 5 GHz Wi-Fi mesh with per-link SSID and channel separation |
| **Virtualized Testbed** | 5 × Linux VMs on Proxmox VE, 3 virtual bridges (vmbr10/20/30), mirrors the Raspberry Pi logical topology |

### Virtualized Topology

```
vm1 (sender)
  ├── enp6s19 (┊ vmbr10 ┊ 10.20.20.0/24) ╌╌╌► vm2 ╌╌╌► vm5   [Path 1 / port 5301]
  ├── enp6s20 (┊ vmbr20 ┊ 10.30.30.0/24) ╌╌╌► vm3 ╌╌╌► vm5   [Path 2 / port 5302]
  └── enp6s21 (┊ vmbr30 ┊ 10.40.40.0/24) ╌╌╌► vm4 ╌╌╌► vm5   [Path 3 / port 5303]
                                                         ↓
                                               dummy0: 10.0.0.5/32
```

All three paths converge at a **dummy interface** (`10.0.0.5`) on vm5 — providing a single logical destination for all flows while keeping forwarding paths physically separate.

## Deterministic Path Selection

Path selection is implemented entirely in the Linux kernel using **DSCP-based policy routing**. No application-layer changes are required.

```bash
# --- vm5: unified dummy destination ---
ip link add dummy0 type dummy
ip addr add 10.0.0.5/32 dev dummy0
ip link set dummy0 up

# --- vm1: per-path routing tables ---
ip rule add tos 0x10 lookup path_1   # port 5301 → vm2 (10.20.20.12)
ip rule add tos 0x20 lookup path_2   # port 5302 → vm3 (10.30.30.12)
ip rule add tos 0x30 lookup path_3   # port 5303 → vm4 (10.40.40.12)

# --- vm1: DSCP marking by destination port ---
iptables -t mangle -A OUTPUT -p udp --dport 5301 -j DSCP --set-dscp 0x10
iptables -t mangle -A OUTPUT -p udp --dport 5302 -j DSCP --set-dscp 0x20
iptables -t mangle -A OUTPUT -p udp --dport 5303 -j DSCP --set-dscp 0x30
```

### Path Validation (Traceroute)

The routing design was verified with `traceroute -U` (UDP) and `traceroute -T` (TCP) probes on all three ports before conducting experiments.

| Protocol | Port | First-hop IP | Destination | RTT range (ms) |
|---|---|---|---|---|
| UDP | 5301 | 10.10.10.162 | 10.0.0.5 | 1.07 – 1.40 |
| UDP | 5302 | 10.10.10.163 | 10.0.0.5 | 0.86 – 1.53 |
| UDP | 5303 | 10.10.10.164 | 10.0.0.5 | 1.17 – 1.74 |
| TCP | 5301 | 10.10.10.162 | 10.0.0.5 | 1.05 – 1.72 |
| TCP | 5302 | 10.10.10.163 | 10.0.0.5 | 0.66 – 1.23 |
| TCP | 5303 | 10.10.10.164 | 10.0.0.5 | 0.73 – 1.16 |

Each port reaches a different first-hop node for both UDP and TCP, confirming correct deterministic forwarding.

## Transport Framework

### Transmission Modes

| Mode | Protocol | Distribution strategy |
|---|---|---|
| `single` | TCP / UDP / FEC | One flow on port 5301 |
| `multi` | TCP | File split into 3 contiguous segments, one per flow |
| `multi` | UDP / FEC | Packets distributed round-robin across all 3 ports |

### XOR-based UDP FEC

Every `k` consecutive source packets form one coding block. A single repair (parity) packet is generated by byte-wise XOR:

```python
for each coding block of k packets:
    parity = p1 XOR p2 XOR ... XOR pk
    transmit p1 ... pk
    transmit parity
```

At the receiver, if exactly **one** source packet is missing and the parity packet is available, the missing payload is recovered. Redundancy ratio ≈ `1/k`. The block size `k` is configurable to trade off overhead vs. recovery strength.

### Network Emulation

`tc tbf` (rate limiting) and `tc netem loss` (packet loss injection) are applied remotely via SSH to all path interfaces before each experiment and cleared after:

```bash
bash net_control.sh apply 20mbit 0% 20mbit 5% 20mbit 10%  # set rates + loss per path
bash net_control.sh clear                                   # restore default
```

## Experimental Results

All experiments transmit the same 111 MB file (116,389,364 bytes). Each condition is repeated 3 times. Results below show averages.

### TCP: Single-Path vs Multi-Path under Loss

| Rate (per path) | Loss | Mode | Avg. Transfer Time (s) | Avg. Goodput (Mbps) |
|---|---|---|---|---|
| 30 Mbit/s | 0% | single | 32.44 | 28.70 |
| 30 Mbit/s | 0% | multi (3×10) | 32.43 | 28.71 |
| 30 Mbit/s | 3% | single | **71.38** | **13.06** |
| 30 Mbit/s | 3% | multi (3×10) | **40.46** | **23.03** |
| 30 Mbit/s | 5% | single | ≈207 | ≈4.5 |
| 30 Mbit/s | 5% | multi (3×10) | **33.78** | **27.54** |

> **Multi-path TCP at 3% loss is ~1.8× faster** than single-path. At 5% loss, the advantage grows dramatically.

### TCP: Effect of Link Rate (Single-Path, 3% Loss)

| Rate | Avg. Transfer Time (s) | Avg. Goodput (Mbps) |
|---|---|---|
| 10 Mbit/s | 115.6 | 8.06 |
| 20 Mbit/s | 78.1 | 11.86 |
| 50 Mbit/s | 68.2 | 13.74 |
| 20 Mbit/s (multi) | 27.5 | 33.84 |

> Multi-path at 3×20 Mbit/s outperforms single-path at 50 Mbit/s with identical total bandwidth budget.

### UDP vs TCP vs FEC (Zero-Loss Baseline, Multi-Path)

| Protocol | Avg. Goodput (Mbps) | File Complete | Notes |
|---|---|---|---|
| TCP multi | 57.43 | Yes | Reliable, retransmission overhead |
| UDP multi | 57.23 | Partial | High raw throughput; gaps under loss |
| FEC (k=4) multi | 45.05 | Yes (recovered) | Stable under loss; ~21% overhead |

### FEC Recovery under Loss

| Loss | Block k | Recovery Rate | Residual Loss | Block Success |
|---|---|---|---|---|
| 1% | 4 | ~90% | ~0.1% | ~98% |
| 3% | 4 | ~75% | ~0.5% | ~93% |
| 5% | 4 | ~60% | ~1.2% | ~88% |
| 3% | 2 | ~92% | ~0.2% | ~97% |
| 3% | 8 | ~58% | ~0.9% | ~89% |

> Smaller `k` provides stronger single-loss recovery; larger `k` reduces overhead but degrades recovery under bursty loss.

Full raw data is in [`results/`](./results/).

## Quick Start

**Prerequisites**: 5 Linux nodes, Python 3.8+, `tc` (iproute2), `iptables`, `sshpass`

```bash
# Single UDP multi-path transfer
python3 receiver_udp.py multi          # on vm5
python3 sender_udp.py testfile.bin multi  # on vm1

# FEC transfer with block size k=4
python3 receiver_fec.py multi          # on vm5
python3 sender_fec.py testfile.bin multi 4  # on vm1

# Full automated experiment suite → results/experiment_results.csv
python3 run_experiments.py
```
